# 發行說明轉換工具

將內部Sprint發行說明（從Slack `#aem-sites-optimizer-announcements`管道或`.cursor/commands/release-notes`游標輸出）轉換為客戶導向的專案，並將其附加至`help/documentation/release-notes.md`。

## 用途

叫用這項技能，然後在出現提示時貼上內部發行說明內容。 技能會：

1. 請套用下列篩選和色調規則的准則。
2. 剖析內部發行說明（表情符號分類的區段： ✨功能、🚀增強功能、🤖 AI優先、🔧修正、🏢 BackOffice）。
3. 根據指南篩選出所有排除的類別（AI工具、後台、本地化字串、E2E測試、SitesInternal-only專案）。
4. 使用下列音調範例作為參考，以客戶對面的音調重寫剩餘的專案。
5. 依功能區域群組相關專案（而非依團隊或存放庫）。
6. 在下列頁面結構範本後格式化為新發行專案。
7. 將新專案附加至`help/documentation/release-notes.md`前面（位於上一個最新專案上方、頁面介紹段落下方）。
8. 列印摘要表格，其中顯示：保留專案、重新寫入專案、卸除專案（以及每個卸除專案的原因）。

## 准則

### 核心原則

1. **客戶權益優先。** 每個專案都應該回答「我以前做不到的事現在該做什麼，或者做得更好？」  — 不是「我們送的是什麼？」 以值開頭，而非實作。

2. **領導膚色。** 為決策者撰寫：成果和能力，而不是技術機制。 數位體驗的VP應該立即瞭解更新重要的原因。

3. **沒有內部行話。** 取代所有團隊內部速記：
   - 「PLG」→「試用使用者」或「新客戶」
   - 「BackOffice」→完全省略（僅限基礎建設變更）
   - 「MSM」→「AEM多網站管理員」
   - 「SHM」→「網站健康情況監視」
   - &quot;OrcaFix&quot;、&quot;Cursor commands&quot;、&quot;AGENTS.md&quot;→完全省略
   - 「EDS」→「Edge Delivery Services」

4. **個短專案。** 一句&#x200B;*什麼*，一句&#x200B;*為什麼它很重要*。 如果兩個都適合一個句子，請這樣做。

5. **範圍準確。** 僅包含客戶將在產品UI中看到的變更或在其工作流程中的體驗。 不包括基礎結構、工具和開發人員經驗變更。

### 頁面結構範本

每個發行專案都遵循此結構：

```markdown
## [Month Start]–[Day End], [Year]

### New Features

- **[Feature Name]** — [One-sentence benefit statement. One sentence of business context if needed.]

### Enhancements

- **[Enhancement Name]** — [One-sentence improvement statement.]

### Bug Fixes

- [Short description of what was fixed and why it matters to users.]
```

**規則：**
- 日期範圍格式： `May 11–22, 2026` （短破折號、縮寫月份、四位數年份）。
- 反向時間順序：頁面頂端的最新發行版本。
- 僅包含具有內容的區段。 若為空白，請省略「增強功能」或「錯誤修正」。
- 錯誤修正專案不使用粗體功能名稱 — 它們是簡單的專案符號。
- 只有在3個或更多使用者可見的修正值得注意時，才包含錯誤修正。

### 要包含與排除的專案

**包含：**

| 類別 | 範例 |
|---|---|
| 新的機會型別 | 廣告意圖不符，請勿在摺頁上方使用CTA |
| 新檢視或工作流程 | 「已部署」標籤、CSV匯出、Jira連結 |
| 試用/上線改進 | 引導式設定流程，非網站已上線狀態 |
| 設定改善 | 稽核目標URL、傳遞型別設定 |
| 有意義的UX修正 | 不正確的計數、中斷的導覽、影響決策的顯示問題 |
| 新資料/整合 | 有機搜尋中的Ahrefs資料，安全性中的相依性樹狀結構 |
| 部署至作者功能 | 支援直接部署的新機會型別 |

**排除：**

| 類別 | 原因 |
|---|---|
| AI工具（OrcaFix、Cursor指令、AGENTS.md、Claude程式碼規則） | 內部開發人員工具，客戶看不到 |
| 本地化細語/預先提交鉤點 | 工程流程，而非產品功能 |
| 後台/基礎架構變更 | 除非他們變更一般使用者行為，否則在UI中不可見 |
| React Spectrum版本升級 | 內部相依性，使用者看不到 |
| E2E測試改善 | 工程品質，而非產品功能 |
| 發行管道自動化 | 內部程式 |
| Sites僅限內部的功能 | 不提供給客戶 |

### 色調範例

| 內部詞語 | 面向客戶的詞句 |
|---|---|
| 「針對手動驗證工作流程引入REJECTED狀態」 | 「您現在可以將建議標籤為已拒絕，以表示這些建議不適用於您的網站，讓您的機會清單專注於可操作的專案。」 |
| 「Canonical和Hreflang機會的部署檢視（依日期分組）」 | 「Canonical和Hreflang機會的變更現在會依部署日期分組在Deployed索引標籤中，讓您清楚瞭解修正內容和修正時間。」 |
| 「Alt Text Autofix V2 — &#39;檢查修復能力&#39;預檢評估」 | 「在部署替代文字修正之前，您可以執行預檢檢查，以確認修正可成功套用至您的內容。」 |
| 「SHM量度的96%儲存最佳化」 | 省略 — 僅基礎結構 |
| 「AGENTS.md，含正式代理程式角色與安全護欄」 | 省略 — 內部AI工具 |
| 「E2E測試效能最佳化（約6分鐘→約5分鐘）」 | 省略 — 工程流程 |

### 群組規則

- **依功能區域**&#x200B;分組，而非依團隊或存放庫。 例如，所有Alt Text改進（功能、增強功能和修正）都屬於相同區域，不會分散於不同區域。
- **將密切相關的修正專案**&#x200B;合併為單一專案符號，而非分別列出每個專案（例如，「付費流量、協助工具及安全性商機中的多項顯示及版面配置改良」）。
- **錯誤修正區段的臨界值**：只有在有3個或更多使用者可見的修正值得發出時，才包含此區段。 應該忽略低於此臨界值的瑣碎或純粹的裝飾性修正。

## 步驟

1. 套用這個檔案中的准則 — 內部化所有原則，包括/排除規則、色調範例和群組規則。
2. 若尚未提供，請詢問使用者涵蓋的日期範圍（例如「2026年5月11日至22日」）。
3. 要求使用者貼上內部發行說明內容（或接受檔案路徑）。
4. 處理內容：
   - **剖析**&#x200B;每個區段(✨/🚀/🤖/🔧/🏢)及其專案符號。
   - 根據上述「排除」表格&#x200B;**篩選器**。 標示每個掉格專案的原因。
   - **重寫**&#x200B;以客戶語調保留的專案：利益優先，沒有行話，簡短專案。
   - **依功能區域分組**，其中有多個相關專案。
   - **臨界值檢查**：如果有3個以上使用者可見的修正，則僅包含「錯誤修正」區段。
5. 使用上述頁面結構範本格式化新專案。
6. 讀取`help/documentation/release-notes.md`的目前內容。
7. 在頁面簡介段落之後立即插入新專案（在前`##`個日期標題之前）。
8. 寫入更新的檔案。
9. 列印摘要表格。

## 輸入格式

此技能接受標準團隊格式的內部發行說明：

```
*ASO UI Release Notes — [Date Range]*
Collaborators: [teams]

✨ *Features*
• [Feature description]

🚀 *Enhancements*
• [Enhancement description]

🤖 *AI-First Development*
• [AI tooling items — will be dropped]

🔧 *Fixes & UX Improvements*
• [Fix description]

🏢 *BackOffice*
• [BackOffice items — will be dropped]
```

## 輸出

技能輸出：

1. 格式化客戶面對的專案（供撰寫前檢閱）。
2. 修改`release-notes.md`前的確認提示。
3. 寫入後：保留/重寫/捨棄專案的摘要表格。
